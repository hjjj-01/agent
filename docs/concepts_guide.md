## LangChain 简介

### 什么是 LangChain？

LangChain 是一个用于开发由大语言模型（LLM）驱动的应用程序的框架。它提供了一套工具和抽象，让你能够轻松地将LLM与外部数据源、API和计算资源连接起来。

### 核心组件

#### 1. **Model I/O (模型输入输出)**
- **Prompts (提示词)**: 格式化和管理发送给LLM的输入
- **LLMs**: 大语言模型的接口（如OpenAI GPT）
- **Output Parsers**: 解析LLM的输出

#### 2. **Data Connection (数据连接)**
- **Document Loaders**: 从不同来源加载文档
- **Text Splitters**: 将长文档分割成小块
- **Vector Stores**: 向量数据库，用于存储和检索向量化的文本
- **Retrievers**: 检索相关文档的接口

#### 3. **Chains (链)**
将多个组件组合起来，形成完整的处理流程。例如：
- 一个链可能包含：获取用户问题 → 检索相关文档 → 构建提示词 → 调用LLM → 返回答案

#### 4. **Agents (代理)**
Agent是LangChain中最强大的组件，它能够：
- 理解用户意图
- 决定使用哪些工具（Tools）
- 执行多步骤任务
- 根据结果调整下一步行动

## RAG (检索增强生成) 原理

### 什么是 RAG？

RAG 是一种结合了信息检索和文本生成的技术。简单来说：
- **传统LLM**: 只依赖训练时学到的知识
- **RAG**: 先从知识库中检索相关信息，再让LLM基于这些信息生成答案

### RAG 的工作流程

```
用户问题
    ↓
[1] 问题理解与向量化
    ↓
[2] 向量检索（从向量数据库中找到最相关的文档）
    ↓
[3] 构建增强提示词（问题 + 检索到的文档）
    ↓
[4] LLM生成答案（基于检索到的文档）
    ↓
最终答案
```

### 为什么需要 RAG？

1. **知识更新**: LLM的训练数据有截止日期，RAG可以访问最新数据
2. **领域专精**: 可以让LLM了解你的业务数据（如WMS系统数据）
3. **减少幻觉**: 提供真实数据作为参考，降低LLM编造内容的风险
4. **数据安全**: 敏感数据可以存储在本地向量数据库中

### 关键技术点

#### 1. **Embeddings (嵌入/向量化)**
将文本转换为数字向量，使计算机能够理解文本的语义含义。
- 相似的文本 → 相似的向量
- 不同的文本 → 差异大的向量

#### 2. **Vector Database (向量数据库)**
专门存储和检索向量的数据库，如：
- **ChromaDB**: 轻量级，适合开发和小型项目
- **FAISS**: Facebook开源，性能优秀
- **Pinecone**: 云服务，生产环境推荐

#### 3. **Chunking (文本分块)**
将长文档分割成小块（chunks），因为：
- LLM有上下文长度限制
- 小块更精确，检索更准确
- 常见策略：按字符数、段落、语义分割

## Agent 工作原理

### 什么是 Agent？

Agent 是一个能够**自主决策**和**执行任务**的AI系统。与传统的"单次问答"不同，Agent可以：
- 将复杂任务分解为多个步骤
- 决定每一步使用什么工具
- 根据中间结果调整策略
- 完成多轮交互直到达成目标

### Agent 的核心概念

#### 1. **Tools (工具)**
Agent 可以调用的功能单元。例如：
- **WMS查询工具**: 查询库存信息
- **计算器工具**: 执行数学计算
- **搜索工具**: 搜索知识库
- **API调用工具**: 调用外部服务

#### 2. **Tool Calling (工具调用)**
Agent 决定何时、如何调用工具的过程：
```
用户: "帮我查一下SKU001的库存，并计算如果订货100件需要多少钱？"
    ↓
Agent思考:
  1. 需要查询库存 → 调用WMS查询工具
  2. 需要计算总价 → 调用计算器工具
    ↓
执行工具调用 → 返回结果
```

#### 3. **Reasoning (推理)**
Agent 使用 LLM 来：
- 理解用户意图
- 分析可用工具
- 制定执行计划
- 解释执行结果

### Agent 的工作流程（ReAct模式）

ReAct = Reasoning + Acting（推理 + 行动）

```
循环执行：
1. Thought (思考): 分析当前状态，决定下一步
2. Action (行动): 选择并执行工具
3. Observation (观察): 获取工具执行结果
4. ... 重复直到任务完成

最终输出: 最终答案
```

### 示例：查询库存场景

```
用户: "SKU001现在有多少库存？是否需要补货？"

第1轮：
Thought: 需要查询SKU001的当前库存
Action: use_tool("wms_query", {"sku": "SKU001"})
Observation: 当前库存50件，安全库存100件

第2轮：
Thought: 库存低于安全库存，需要补货
Action: use_tool("calculator", {"operation": "subtract", "a": 100, "b": 50})
Observation: 需要补货50件

第3轮：
Thought: 已经有足够信息回答用户
Final Answer: SKU001当前库存50件，低于安全库存100件，建议补货50件。
```

### 数据流示例

#### 场景1：查询库存
```
用户问题: "SKU001有多少库存？"
    ↓
飞书机器人接收消息
    ↓
Agent分析意图 → 识别为"库存查询"
    ↓
调用WMS工具 → wms_query_inventory(sku="SKU001")
    ↓
WMS API返回数据 → {"sku": "SKU001", "quantity": 50}
    ↓
Agent生成答案 → "SKU001当前库存为50件"
    ↓
飞书机器人返回消息给用户
```

#### 场景2：基于知识的问答
```
用户问题: "如何处理退货订单？"
    ↓
Agent分析意图 → 识别为"知识查询"
    ↓
调用RAG工具 → rag_search(query="退货订单处理流程")
    ↓
向量数据库检索 → 返回操作手册相关章节
    ↓
Agent基于检索内容生成答案 → "根据操作手册，退货订单处理流程如下：1. ..."
    ↓
飞书机器人返回消息给用户
```

### 模块职责划分

#### 1. **src/wms/** - WMS接口模块
- **wms_client.py**: 封装WMS API调用
- **models.py**: WMS数据模型定义
- **功能**:
  - 调用WMS系统接口
  - 数据格式转换
  - 错误处理

#### 2. **src/rag/** - RAG知识库模块
- **vector_store.py**: 向量数据库管理
- **document_processor.py**: 文档处理和向量化
- **retriever.py**: 检索器
- **功能**:
  - 文档加载和分割
  - 文本向量化
  - 相似度检索
  - 知识库管理

#### 3. **src/feishu/** - 飞书机器人模块
- **bot.py**: 飞书机器人核心类
- **message_handler.py**: 消息处理逻辑
- **server.py**: Web服务器
- **功能**:
  - 接收飞书消息
  - 消息验证和解析
  - 发送回复消息
  - 事件处理

#### 4. **src/agent/** - Agent核心模块
- **wms_agent.py**: Agent主类
- **tools.py**: 工具定义
- **prompts.py**: 提示词模板
- **功能**:
  - Agent初始化和配置
  - 工具注册和管理
  - 对话流程控制
  - 上下文管理

#### 5. **src/utils/** - 工具模块
- **config.py**: 配置管理
- **logger.py**: 日志系统
- **helpers.py**: 辅助函数
- **功能**:
  - 配置加载和验证
  - 日志记录
  - 通用工具函数
