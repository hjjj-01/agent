## Bug 修复：query_inventory 崩溃（TypeError: argument after ** must be a mapping, not InventoryQueryRequest）

### 现象
运行时报错：`InventoryQueryRequest() argument after ** must be a mapping, not InventoryQueryRequest`
- 触发位置：`wms_client.py:query_inventory` 第61行 `request_model = InventoryQueryRequest(**params)`

### 根因
重构时 `tools.py` 的 `query_inventory` 把 `params` 构造成了 **InventoryQueryRequest 实例** 并传给 `wms_client.query_inventory(params)`；
而 `wms_client` 又对该实例再次执行 `InventoryQueryRequest(**params)` —— 把模型实例当 dict 解包，导致崩溃。
（原作业量代码里 tools 传的是 dict，wms_client 再包一层模型，两层约定不一致。）

### 修复
`wms_client.py` 的 `query_inventory` 直接接收 InventoryQueryRequest 实例，改用 `payload = params.model_dump(exclude_none=True)`，不再二次解包。
`tools.py` 保持构造模型实例的写法不变。

### 验证
- 复现组合查询（库位+SKU）路径，不再崩溃；格式化输出正常
- 实际发给 API 的 payload 含 locationCode/itemLabel/itemValue/filterNonZero=Y/isIncludeVirtual=N，与用户确认可用的请求一致
- 仅改动 `wms_client.py` 一处，已 py_compile 通过

## Bug 修复：_make_request 中 response 变量未绑定（NameError 掩盖真实错误）

### 现象
配置错误（服务器 .env 仍是占位符 your-wms-api.com）导致 DNS 解析失败后，
报错 `cannot access local variable 'response' where it is not associated with a value`，
真实网络错误被掩盖。

### 根因
`_make_request` 的 `response` 变量只在 try 内的 `response = self.session.request(...)` 赋值；
当该赋值语句本身抛 RequestException 时，except 块里的 `if response is not None` 引用了从未绑定过的局部变量 → UnboundLocalError。

### 修复
在 try 之前先 `response = None` 初始化，确保 except 中引用安全。
（仅改 `wms_client.py` 一处）

### 真正的根因（需用户在服务端处理）
- 本地 `.env` 的 `WMS_API_BASE_URL=https://wms.lasenyun.com/admin-api` 是正确的；
  但**服务器上的 .env 仍是占位符 `your-wms-api.com`**，需把服务器的 `WMS_API_BASE_URL` 改成 `https://wms.lasenyun.com/admin-api`。
- 另：本地 `.env` 的 `WMS_API_TOKEN=your-wms-token-here` 也是占位符，服务器需填入真实 token（先修复 DNS 才会走到鉴权）。
- 错误 URL 还显示路径前缀 `/api/`，说明服务器旧 .env 的 base 可能是 `http://your-wms-api.com/api`。
  正确 base 应为 `https://wms.lasenyun.com/admin-api`（无 /api 前缀，endpoint 自带 `wms/...`）。
